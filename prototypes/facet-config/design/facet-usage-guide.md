# Facet Config: How It Works

A guide to the facet-based config system prototyped in `prototypes/facet-config/`,
for developers who know serde and the existing `define_tedge_config!` macro.

The prototype uses facet `0.50.0-rc.5` with the safe `facet-reflect` API
(`Peek`/`Partial`) throughout — there is no unsafe code in the runtime.

## Mental model

The existing macro generates *behaviour*: match arms, key enums, and
parse/display glue for every field, for every operation. That's why it's
~4,400 lines and why extending it is daunting.

The facet design splits the problem differently:

- **The macro generates data only.** `define_config!` emits two plain struct
  trees (a serde-facing DTO and a typed Reader) plus five small registry
  functions. No match arms, no key enums, no per-key logic.
- **All behaviour is generic.** get, set, unset, add, remove, list, env-var
  overlay, and reader construction are each written once in `config-runtime`,
  as functions that walk any config struct's facet `Shape` at runtime.

serde's role is unchanged: DTOs derive `Serialize`/`Deserialize` and all TOML
file I/O goes through `toml`, exactly as today. Facet handles only the
"dotted key ↔ struct field" mapping.

| Concern | Handled by |
|---|---|
| TOML read/write | serde + `toml` (as today) |
| Key traversal, `FromStr` parsing, `Display`, key listing | facet reflection (generic code in `config-runtime`) |
| Defaults, aliases, read-only keys, examples | registries generated by the macro (plain data) |

## Defining a config

```rust
facet_config_macro::define_config! {
    TEdge {
        device: {
            /// Unique device identifier
            #[tedge_config(example = "my-device-001")]
            id: String,

            /// Device type identifier
            #[tedge_config(rename = "type", default(value = "thin-edge.io"))]
            ty: String,

            /// Path to the device certificate
            #[tedge_config(default(from_config_dir = "device-certs/tedge-certificate.pem"))]
            cert_path: camino::Utf8PathBuf,
        },
        mqtt: {
            /// MQTT broker port
            #[tedge_config(default(value = "1883"), deprecated_key = "mqtt.external.port")]
            port: u16,
        },
    }
}
```

Groups nest arbitrarily (`ident: { ... }`); leaves are `ident: Type`; a schema
defined by another `define_config!` invocation mounts as
`ident: extern path::To::ReaderType` (see "Shared schemas"). Doc comments
become the help text for `list` and shell completions.

## What the macro generates

See `crates/config-macro-impl/src/{dto,reader,schema}.rs`, all walking the
intermediate model built in `model.rs` (which owns every computed struct name
and dotted key). For the example above:

**1. DTO structs** — `TEdgeConfigDto`, `DeviceConfigDto`, `MqttConfigDto`.
Every leaf is `Option<T>`, every group field is `Option<GroupDto>`:

```rust
#[derive(Debug, Default, Clone, ::facet::Facet, ::serde::Serialize, ::serde::Deserialize)]
#[facet(type_tag = "config_group")]
pub struct MqttConfigDto {
    /// MQTT broker port
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub port: Option<u16>,
}
```

This is what serde reads from / writes to TOML. `skip_serializing_if` keeps
unset keys and empty groups out of the file. The `type_tag` marks the struct
as a group the runtime may recurse into — as opposed to a leaf like
`HostPort`, which is also a struct but must be treated as a single value.

**2. Reader structs** — `TEdgeConfig`, `DeviceConfig`, `MqttConfig`. A field
with a concrete default (`value`, `function`, `from_config_dir`, `from_key`)
becomes plain `T`; anything else stays `Option<T>` — including `from_root` and
`from_optional_key`, whose fallbacks can legitimately be absent:

```rust
#[derive(Debug, ::facet::Facet)]
#[facet(type_tag = "config_group")]
pub struct MqttConfig {
    pub port: u16,   // default(value = "1883") ⇒ always present
}
```

Readers derive only `Facet` (they are never serialized) and are populated by
one generic function, `build_reader` — not per-field codegen.

**3. A `ConfigSchema` impl** — one trait impl on the reader type, linking it
to the DTO type and exposing the schema's metadata as plain data:

| Item | Contents |
|---|---|
| `type Dto` | the matching DTO type (`TEdgeConfigDto`) |
| `defaults(config_dir)` | `Vec<FieldDefault>`: dotted key → `DefaultSpec` |
| `read_only_keys()` | dotted keys that refuse set/unset/add/remove |
| `aliases()` | deprecated key → canonical key pairs |
| `examples()` | key → example strings |
| `register_types(registry)` | one `register_append_remove::<T>()` per distinct leaf type |

Because everything hangs off the reader type, several `define_config!`
invocations can share a module without colliding.
`ConfigManager::from_schema::<TEdgeConfig>(config_dir)`
(`crates/config-runtime/src/manager.rs`) builds the validated registries from
this data and is the main runtime handle.

## Attribute reference

Everything lives under `#[tedge_config(...)]`, parsed by the macro itself
(via darling). Facet custom attributes are not used.

| Attribute | On | Effect |
|---|---|---|
| `rename = "type"` | field | `#[facet(rename)]` + `#[serde(rename)]` on the DTO field, so the CLI key and TOML key both use the new name |
| `readonly` | field | key goes into `ReadOnlyKeys`; set/unset/add/remove refuse with `ConfigError::ReadOnly` |
| `deprecated_key = "old.key"` | field | entry in `KeyAliases`; the old key still resolves, with a deprecation warning |
| `example = "..."` (repeatable) | field | collected into `build_examples()`; shown by `list` and completions |
| `default(value = "1883")` | field | `DefaultSpec::Value`; reader field is concrete `T` |
| `default(function = path::to::fn)` | field | `DefaultSpec::Function(fn() -> String)`; concrete reader field |
| `default(from_config_dir = "rel/path")` | field | joined onto the config dir when the registry is built; concrete reader field |
| `default(from_key = "other.key")` | field | falls back to another key, which must itself resolve (validated at startup); concrete reader field |
| `default(from_optional_key = "other.key")` | field | falls back only if the other key is set; reader field stays `Option<T>` |
| `default(from_root = "device.cert_path")` | field | mapper config falls back to the root `tedge.toml`; reader field stays `Option<T>` |

Groups and `extern` mounts accept no `tedge_config` attributes; any present
are rejected at compile time naming the unknown option.

Notes:
- All default values are **strings**, parsed through the same `FromStr` path
  as `config set`: `default(value = "1883")`, never a typed literal. This
  replaces the existing macro's `value`/`variable`/`from_str` distinctions.
- `///` doc comments replace the old `note` attribute; facet exposes them at
  runtime as `field.doc`.

## How the operations work

All generic over `T: Facet`, in `crates/config-runtime/src/reflect.rs`.

**get** — `config_get(dto, "mqtt.port")` wraps the DTO in a `Peek`, splits the
key on `.`, walks struct fields (matching `field.rename` falling back to
`field.name`), transparently unwraps `Option`s, and `Display`-formats the
leaf. Lists format as comma-joined items.

**set / unset** — `config_set` rebuilds the whole DTO through
`Partial::alloc::<T>()`: it copies every existing field, and at the target key
calls `partial.parse_from_str(value)` — facet's type-erased `FromStr`
dispatch — wrapped in `begin_some()` for `Option` fields. `unset` writes
`None` via `set_default()`. Absent intermediate groups (`mqtt = None`) are
materialized along the way. Rebuilding per write is deliberate: it keeps the
code entirely safe, and config writes are not a hot path.

**add / remove** — string-level operations layered on get/set. The current
value and the new item go through an `AppendRemoveItem` implementation, looked
up in the `AppendRemoveRegistry` by the leaf type's `ConstTypeId`:

```rust
pub trait AppendRemoveItem: Sized {
    fn append(current: Option<Self>, new_value: Self) -> Option<Self>;
    fn remove(current: Option<Self>, remove_value: Self) -> Option<Self>;
}
```

Scalars replace on add and clear-if-equal on remove; `TemplatesSet` does set
union / difference. The registry is generic infrastructure
(`type_action.rs`) — adding a new kind of type-keyed operation is one macro
call plus one generated registration line.

**list / completions** — a recursive `Shape` walk emits
`KeyEntry { key, doc, examples }` for every leaf, skipping `Def::Map` fields
(so profile maps never appear as settable keys). `doc` comes straight from
facet's doc-comment reflection.

**typed reader** — `build_reader::<Dto, Reader>(dto, defaults, root_resolver)`
walks the *Reader's* shape: a required field must resolve from the DTO or a
default (otherwise a "Required config key … not set" error); optional fields
become `Some`/`None`. `build_reader_with_root` wires in a resolver for
`from_root` defaults.

## Defaults

`DefaultsRegistry` (`defaults.rs`) maps dotted keys to:

```rust
pub enum DefaultSpec {
    Value(String),                      // default(value), default(from_config_dir)
    Function(fn() -> String),           // default(function)
    FromKey(Cow<'static, str>),         // default(from_key)
    FromOptionalKey(Cow<'static, str>), // default(from_optional_key)
    FromKeyVia { .. },                  // default(from_key_via)
    FromRoot(&'static str),             // default(from_root)
}
```

Same-schema source keys (`FromKey`, `FromOptionalKey`, `FromKeyVia`) are
`Cow<'static, str>` because mounting a shared schema rewrites them into the
mounting schema's key space; `FromRoot` names a key of the root config, which
no mount ever rewrites, so it stays `&'static str`.

The registry validates itself on construction (inside
`ConfigManager::from_schema`): a `from_key` chain that cannot terminate in a
real value (or exceeds the cycle-guard depth) fails at startup. Validation
runs on the fully mounted key set, so a dangling reference inside a shared
schema is caught wherever that schema is used. This is the runtime equivalent
of the old macro's compile-time guarantee — equivalent in practice because
tests and production construct the registry through the same entry point.

## Shared schemas (`extern` groups)

A schema can mount another `define_config!` schema under one of its keys:

```rust
facet_config_macro::define_config! {
    MapperDevice {
        /// Unique device identifier for this mapper
        #[tedge_config(default(from_key_via(key = "cert_path", function = "certificate_common_name")))]
        id: String,

        /// Path to the device certificate for this mapper
        #[tedge_config(default(from_root = "device.cert_path"))]
        cert_path: camino::Utf8PathBuf,
    }
}

facet_config_macro::define_config! {
    C8yMapper {
        url: HostPort<HTTPS_PORT>,

        /// Identity of the device this mapper connects to Cumulocity
        device: extern MapperDeviceConfig,
    }
}
```

The `extern` type is the mounted schema's *reader* type — any path works, and
a type that was not generated by `define_config!` fails to compile with a
missing `ConfigSchema` impl reported against the `extern` line. The generated
DTO embeds `Option<<MapperDeviceConfig as ConfigSchema>::Dto>`, the reader
embeds `MapperDeviceConfig` directly, and the mounting schema's `ConfigSchema`
impl extends its own metadata with the mounted schema's, remapped under the
mount key by the `prefix_*` helpers in `config-runtime/src/schema.rs`.

Key-space rules inside a shared schema:

- `from_key`/`from_optional_key`/`from_key_via` name keys of the shared
  schema itself (`cert_path` above) and are remapped at the mount point
  (`device.cert_path` when mounted at `device`), so the schema works at any
  mount key.
- `from_root` names a key of the root tedge config and is never remapped.
- There is currently no way for a shared schema to reference a key of the
  schema that mounts it. A survey of the production `tedge_config.rs` found
  no defaults that would need this; if it ever comes up, a marked absolute
  reference (e.g. a `^`-prefixed key) can be added without disturbing the
  relative-by-default semantics.

`readonly`, `deprecated_key`, and `example` attributes inside the shared
schema are remapped the same way, so a read-only key stays read-only at its
mounted location and aliases resolve under the mount prefix.

## Environment variables

`EnvOverrides` (`defaults.rs`) applies env vars onto the DTO after loading,
before any effective read. Consequently, `get`, `list`, and `show` report the
same environment-overlaid values that an application sees:

- Prefix-stripped and lowercased: `TEDGE_MQTT_PORT` → `mqtt.port`. The
  underscore/dot ambiguity (`c8y_proxy_bind_port`) is resolved by trying
  splits against the actual key set.
- An empty value means unset. Unknown keys warn, with an exclusion list so
  e.g. `TEDGE_CONFIG_DIR` doesn't trigger one.
- Mapper configs accept `TEDGE_MAPPERS_<NAME>_*`, plus cloud shorthands:
  `TEDGE_C8Y_URL` (applies when no profile is selected) and
  `TEDGE_C8Y_PROFILES_PROD_URL` (applies under `--profile prod`).

The persistent commands deliberately do not use that overlaid DTO. `set`,
`unset`, `add`, and `remove` reload the DTO from TOML, apply the action, and
save it. This prevents unrelated overrides (including empty overrides) from
being persisted and ensures `add`/`remove` operate on the persisted value
rather than an environment-provided value. A later `get` still applies the
environment override, so its output can differ from the raw TOML by design.

## Multiple files (federated config)

`FederatedConfig` (`federated.rs`) mounts several `ConfigOps` sources under
key prefixes — the root `tedge.toml` at `""`, each mapper at
`mappers.<name>.` — and routes every operation to the longest matching prefix.
Each mount is a `TypedConfigOps<Dto>`: its own file, its own DTO type, its own
`ConfigManager`, all sharing the same generic machinery. The CLI adds a
compatibility alias layer on top (`c8y.url` → `mappers.c8y.url`;
`--profile prod` → `mappers.c8y.prod.url`).

`from_root` reads currently work via a fallback scan in
`FederatedConfig::read`; `prototypes/facet-config/design/federated-read-with-root.md`
proposes pushing this into a `ConfigOps::read_with_root` method (not yet
implemented).

## Custom field types

A leaf type needs `FromStr` + `Display` + `Facet`, plus serde attributes for
its file representation. `HostPort<const P: u16>` shows the pattern:

```rust
#[derive(Debug, Clone, PartialEq, Eq, Facet, Serialize, Deserialize)]
#[facet(traits(FromStr, Display, Debug, Clone, PartialEq, Eq))]
#[serde(into = "String", try_from = "String")]
pub struct HostPort<const P: u16> { /* input, host, port */ }
```

`#[facet(traits(…))]` populates the shape's vtable so the generic
`parse_from_str`/`Display` machinery works on the type. Const generics are
fine — each monomorphization gets its own `Shape`. `TemplatesSet` additionally
uses a serde `untagged` wrapper so TOML accepts either an array or a
comma-separated string.

## Gotchas

- Reader fields backed by `from_root`/`from_optional_key` are `Option<T>` —
  only `value`/`function`/`from_config_dir`/`from_key` defaults make a field
  concrete.
- Profiles are realized as separate mount directories
  (`mappers/c8y.prod/mapper.toml`), not as a `HashMap` within one file; the
  runtime's `overlay_dto` supports layering DTOs but no macro syntax uses it.
- `build_reader` treats reflection errors on optional fields as `None`.
- Keys match on the *renamed* field name (`device.type`, not `device.ty`) —
  the macro applies the rename to both facet and serde so the CLI and TOML
  agree.
